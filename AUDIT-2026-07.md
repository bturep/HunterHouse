# Full-project audit — Hunter House Foundation archive

**Date:** 2026-07-02 · **Scope:** security, robustness/preservation, content presentation, peer cross-reference.
Live `browse.html` v1.08.04 · staging `next.html` v1.09-test.74 · catalogue **253 items** (grown past the 226 the docs cite: +EGC photos, +45 IHC). Method: four parallel investigations, each verifying claims against live code/URLs rather than trusting the docs.

This began as a read-only audit; the §A security findings were subsequently remediated on 2026-07-02 (see the project `CLAUDE.md` session log). **This is a redacted copy** — specific exposure identifiers (item IDs, exact object URLs, the commit that held the text) are omitted, since the repo is public.

## A. Private correspondence exposure — found and remediated 2026-07-02

The gated-correspondence architecture (Cloudflare Worker + a private R2 bucket) is soundly built and holds for the production correspondence. The audit found that a small number of **demo** correspondence assets, from a development tryout, had briefly escaped that architecture via three vectors. **All three were remediated on 2026-07-02** (see the project `CLAUDE.md` session log). Specific identifiers — item IDs, exact object URLs, the commit that held the text — are omitted from this committed copy.

### A1. Transcript text had entered git history — CRITICAL — REMEDIATED
- A development tryout briefly committed demo transcript text into `next.html`; it was later removed from the working tree but remained recoverable from history until purged.
- **Remediation:** the text was excised from all history with `git filter-repo` (verified: zero residue across every historical revision) and force-pushed; a mirror backup was retained. Caveat: a public host can still cache pre-rewrite objects until it garbage-collects — request GC for full removal.

### A2. Demo assets were reachable in the public bucket — CRITICAL — REMEDIATED
- A few demo correspondence assets had been placed in the *public* image bucket rather than the private gated bucket, so they were fetchable without auth (and, being public URLs, not protected by the Worker gate).
- **Remediation (Mac-side):** the assets are removed from the public bucket / relocated to the gated bucket, with a CDN cache purge. Standing rule adopted: correspondence assets never go in the public bucket.

### A3. The public bucket browser could enumerate the correspondence prefix — LOW — REMEDIATED
- `/list`'s Origin check is browser-only (forgeable from a non-browser client), so the correspondence prefix could be enumerated.
- **Remediation:** the prefix was added to the Worker's `HIDDEN_PREFIXES` (server-side filtering), independent of the bucket cleanup.

Also from the security pass (lower severity):
- **M1 — shared researcher token could leak via the URL.** One shared token; `?researcher=<token>` was not stripped from the URL after being read into storage, so it could survive into `?id=` share links and `?t=` image URLs. **Partially remediated:** the token is now stripped from the address bar immediately after storage, and an explicit `referrer` policy was added. Still recommended: per-researcher tokens (revocable, attributable) and a token rotation.
- **L2 — gated images cached ~10 min** with the token in the URL; consider `no-store` if shared-machine use is plausible.
- **L4 — `forms/explore-3d.html` loads three.js from a CDN with no SRI** (`:244`); pin SRI hashes or vendor locally (shares the `hunterhouse.org` origin).

**Clean, verified:** the gated token check (constant-time, before routing; path-traversal blocked); `/dl` is not an open redirect; `edit_proxy.py` defences match the docs and hold; no hardcoded credentials anywhere in `scripts/`/`forms/`/`.github/`; the XSS surface is escaped consistently (`escapeHTML`, `transcriptToHTML`, `build_item_pages.py`'s `esc()`); `sw.js` never caches off-origin/gated responses.

## B. Robustness & preservation — the masters are a real single point of failure

### B1. Master TIFs live in exactly one R2 bucket, no versioning — and corruption already happened — CRITICAL
- Commit `564f6ff` (2026-06-30): an unchecked `rclone moveto` collision overwrote **3 original masters in place**. Recovery was possible only because source JPEGs happened to survive in an unmanaged `~/Downloads` folder — not a backup.
- R2 has no object versioning; nothing in `scripts/` copies R2 anywhere; the metadata pipeline backs up Wikibase *to* R2 but nothing backs up R2. Cloudflare account loss takes the masters, the analytics Worker, and the CDN domain at once.
- **Fix (cheap, high-value):** monthly `rclone sync hh-r2:hunter-house-archive /Volumes/<disk>/hh-r2-mirror --backup-dir .../hh-r2-deleted-YYYYMMDD` to one external disk kept off-site. The `hh-r2` remote already exists; `--backup-dir` makes the mirror deletion-safe. Standing rule: any script issuing `rclone moveto`/`deletefile` does a collision dry-run + post-hash check.
- Comforting asymmetry worth stating in a preservation note: the *analog* originals are professionally held at U Calgary, so digital loss is a re-digitization cost, not a history loss.

### B2. Wikibase disaster-recovery sidecars are silently ~6 weeks stale with 404 holes — HIGH (live now)
- `_wikibase/_manifest.json` last modified **2026-05-23**. `HH-EGC-0032.json` / `0059.json` (the whole June 30 photo batch) → **404**. `HH-EGC-0023.json` → 200 but stale despite two relabels this week.
- Root cause: "fail-safe" sidecar syncs are wrapped in `except Exception` with `check=False` (`batch_ingest_egc_photos.py:385`, `renumber_egc_photos.py:146`) — failures are swallowed silently. `verify_r2_links.py` would have caught every 404 but evidently hasn't been run since before June 30.
- **Fix:** `verify_r2_links.py` needs no credentials — put it in a **weekly scheduled GitHub Action that fails loudly** on any non-2xx. Then run `backup_metadata.py` + `sync_metadata_to_r2.py --execute` now to close the gap.

### B3. Zero behavioral test coverage — the Playwright suite was deleted — HIGH
- `tests/` is gone (commit `955436c`); ARCHITECTURE.md §8 still describes three smoke tests. CI checks only: JS parses, VERSION pattern + bump, manifests parse. It cannot catch a runtime error after parse, a broken load ladder, a stale snapshot, a promotion mistake, a CSP regression, or a `sw.js` syntax error (sw.js isn't even `node --check`ed).
- **Best safety-per-effort for a solo maintainer:** one scheduled weekly *health* workflow (no secrets) that watches the **deployed** system — curl `browse.html`, assert VERSION present; fetch `catalogue.json`, assert `generated` < 14 days and `item_count` == live SPARQL count; HEAD-check one image + one sidecar per collection (or just run `verify_r2_links.py`, merging with B2). More valuable than resurrecting local-only Playwright.

### B4. localStorage cache keys accumulate forever → quota → silent note loss — MEDIUM-HIGH (new)
- `CACHE_KEY = "hhf_" + VERSION`; nothing ever deletes old `hhf_*` keys. `next.html` has bumped VERSION 74× this line, each orphaning a ~300 KB entry against a ~5 MB origin quota. When quota hits, `rnSave()` **silently swallows** the QuotaExceededError (`browse.html:4019`) — a researcher's note edit is dropped with no feedback.
- **Fix:** ~6 lines at boot to evict every `hhf_v*` key that isn't the current `CACHE_KEY`; make `rnSave` toast on catch (notes are the one user-authored datum on the site).

### B5. Promotion is a manual ritual that has already misfired twice — MEDIUM
- The v1.08 promotion shipped the staging manifest to live (`857053d`) *and* staging title/icons to live (`c2a4a30`). No drift-diff tooling between the two ~9K-line near-duplicates.
- **Fix:** encode promotion invariants as ~10 lines in `validate.mjs` (already runs every push): `browse.html` must not contain `manifest.next.json`, must not contain `-test.` in VERSION, must reference live icon/title strings, must retain `LIVE GATE` markers if gated features haven't shipped. Turns every historical mistake into a CI email.

### B6. AbortSignal gap still present on all three network paths — MEDIUM-LOW
- No timeouts anywhere. Tier 3 (R2 snapshot) is awaited before first paint for every first-time visitor, so a *stalled* (not failed) R2 connection hangs them on the load screen with a healthy Wikibase never consulted. Add `AbortSignal.timeout(4000)` to `fetchSnapshot`, `AbortSignal.timeout(45000)` to `sparqlQuery` (survives the documented cold start).

### B7. Analytics rollup silent-failure modes — LOW
- GitHub disables cron workflows after 60 days of repo inactivity — exactly the "attention lapses" scenario the archive is built to survive. And `rollup_analytics.py` happily writes an all-zeros month. Add `sys.exit(1)` when `totals.events == 0`; note the 60-day cron rule in the session-log ritual.

### Minor traps
- `validate.mjs:30` staging regex rejects `test.100` (`\d{2}` → `\d{2,}`); staging is at test.74.
- Nothing checks that an `assets/` change (shared `light.css`, patched PDF.js) came with a `CACHE_NAME` bump.
- Curations changed shape (`*.md` + `index.json`); nothing validates them; ARCHITECTURE §3 still says two JSON files.
- `build_item_pages.py` has no orphan pruning — a renumbered item's old page/sitemap entry lingers until manually removed (in sync today by discipline, not construction).

**Verified solid:** the four-tier load ladder matches the docs exactly, every tier's failure is caught, all-tiers-fail shows a readable error (not a hang); snapshot/pages/sitemap are all in sync at 253 today; query single-sourcing holds by construction; CI VERSION-bump guard works; `sw.js` read-path model is conservative and correct.

## C. Content presentation — the model knows more than the published pages say

The intellectual model (creation-vs-digitization dates, temporal authorship roles, provenance-as-prose, Wikidata anchoring, per-item permalinks) is **stronger than most small institutional archives**. Nearly every gap is a *last-mile publication* gap: the model captures things (P100 curatorial notes, rights, ID permanence, phase membership, citation strings) that the pages a scholar actually lands on don't yet display.

**Already strong:** provenance told honestly per collection (`archive.html`); creation vs digitization dates correctly separated (`WIKIBASE.md` P82/P64 vs P83); temporal authorship ("Designed by → Built by → Drawn by" in event order); 253 stable permalinks with schema.org JSON-LD + Wikidata `sameAs`; a fonds-level Wikidata item (Q139960001); the CAA connection presented at three levels; candid editorial voice; a citation apparatus already built in staging (`citeFor()`).

**Gaps, ranked by scholarly impact:**

1. **Rights absent or unresolvable for ~40% of items, and the contact path dead-ends.** 102 of 253 pages have no Rights row (all EGC, all IHC); where rights exist they say "contact the Foundation," and `about.html` says contact addresses "will be published… as channels are established." A scholar can't complete a permissions request. → Put one monitored email on `about.html`; assign every collection a default **RightsStatements.org URI** in Wikibase so nothing renders rights-less; drop the non-standard "Public — HHF license" label.
2. **No recommended-citation block on the static item pages** — the pages Google/Zotero/footnotes actually land on. `citeFor()` exists but only in staging. → Have `build_item_pages.py` emit a one-line Chicago citation per page. Highest scholarly value per line of code.
3. **Extent/scope/ID-permanence never stated publicly, and public numbers are stale & inconsistent.** `archive.html` says 147 items (actual 253); "GES" for what's called EGC everywhere else; nav badges still read v1.04. → A short "Scope and extent" paragraph, with counts generated from the snapshot rather than hand-written.
4. **No archival hierarchy above "collection."** Phases (P62) are bare labels — no date range, scope note, or public page; no fonds-level description tying HHC+EGC+IHC+FRH together the way F0076 does for Calgary. → Add description + date-range to each phase item; optionally generate `archive/phase/<id>.html` (same generator pattern), completing the fonds→series→item ladder the UI already gestures at.
5. **Missing item elements: dimensions and an interpreted location statement.** No dimensions property (core for drawings — DACS 2.5); physical location renders as raw codes ("S0004, SS0001…") with no key, actively confusing on CAA items whose originals are in Calgary. → Mint a dimensions string property, backfill at scan time; caption or suppress location codes; on CAA items state "Original held by CAA, Calgary."
6. **The 253 item-page leads are one mechanical, occasionally ungrammatical template** ("held in Canadian Architectural Archives, 1969-05-01"), and P100 cataloguer's notes exist but aren't on the static pages. → Fix the template; prefer P100 as the lead when present — real prose is also what differentiates the pages in search.
7. **Biography page contradicts itself** — deck says "b. Phoenix AZ 1930," its own chronology says "year pending," Wikidata (yours) says 4 Nov 1930. → Reconcile with the Wikidata item; add one clause clarifying F0076 = accession 2019.61.
8. **Curation format has no date-of-curation**, so a curation can't be cited; the one live curation (`for-theodora.md`) carries zero item notes. → Add a date to the byline convention; add one-sentence notes to Theodora.
9. **Gallery is a discoverability orphan** — linked from no nav, `<title>` just "Hunter House," no route from an image to its catalogue record. → Caption slides with IDs linking to `archive/<ID>.html`; give it a descriptive title.

## D. Peer cross-reference — where Hunter House stands

Benchmarked against CCA (incl. the Erickson fonds), Sir John Soane's Museum, the Frank Lloyd Wright/Avery archive, the Eames Institute, the Paul Rudolph Institute (closest single-architect peer), the Lautner/Getty split, four house museums, Drawing Matter, and the Wikibase-GLAM precedents (Rhizome ArtBase, Enslaved.org, SNAC).

**Differentiators HH already has that most peers lack:**
- Machine-readable item data (JSON-LD + Wikibase Q-item + EntityData JSON + SPARQL) — absent at CCA, Soane, Rudolph, Eames; only Rhizome-class projects match it.
- Full catalogue **and** curator lens together — CCA/Soane are catalogue-only, Eames is curation-only; only Drawing Matter does both, at 100× the size.
- Wikidata `sameAs` linkage; direct public access to master TIFFs (most peers gate high-res behind request forms + fees); **complete** item-level coverage with no "stub vs full record" split; and a single-house archive with its own public catalogue at all — Glass House, Farnsworth, Gropius, Schindler have none.

**Table stakes HH is partial on:** a standard rights statement per item (has free-text custodian, not a RightsStatements.org URI); zoomable images (has preview JPG + master link, no pan/zoom viewer).

**Realistic adoptions, cheapest → richest (with peer precedent):**
1. **RightsStatements.org URIs** (~hours) — the DPLA/Europeana standard for a custodian who isn't the rights holder. Closes gap C1.
2. **Recommended-citation block per item** (~hours) — only LoC among peers offers this, so it's cheap *and* a differentiator. Closes C2.
3. **Preservation ladder** (~a day + habit) — SHA-256 manifest committed to the repo; one off-site external-drive copy; scheduled `rclone check`. Moves HH from ~NDSA Level 1 to solid L2 storage / L3 integrity. Directly implements B1/B2. Publish a one-paragraph preservation statement — peers rarely do; researchers notice.
4. **Sheet/set structure** (~days) — expose sheet n-of-m and drawing-set type (design/working/record), per Soane's "scheme" and Rudolph's set-type sorting; `1of2` is already in filenames.
5. **IIIF level-0** (~a weekend) — static tiles + `info.json` + Presentation manifests generated into the existing R2 bucket, OpenSeadragon inline. Free at 226 items; pencil-on-tracing rewards deep zoom; manifests make drawings citable inside Mirador/scholarly IIIF workspaces.
6. **ARKs** (~days + a persistence promise) — free NAAN; only worth it for decades-scale citation independent of the domain. Canonical URLs + Q-ids already do most of the job.

**Skip:** OAI-PMH (sitemap + JSON-LD + SPARQL already cover discovery), DOIs (cost without benefit at this scale), dynamic image servers (level-0 suffices).
